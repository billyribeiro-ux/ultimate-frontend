# Lesson 10.2 — TypeScript in API routes

## 1. Concept — The three generic slots of `RequestHandler`

### 1.1 Why typing matters even more in handlers

A handler receives data that came over the network. That data is untyped at runtime no matter what you write in TypeScript — the browser or external client can send whatever they like. TypeScript's job in a handler is twofold:

1. **Protect you from mistakes you make** while writing the handler (wrong property names, missing branches).
2. **Document the shape** so callers know what to send and expect.

TypeScript alone cannot enforce the runtime shape. Validation (with Valibot, from Module 9B) is still mandatory. The types and the validator work together: the validator checks the data, the types describe the result of that check.

### 1.2 The three generics of `RequestHandler`

`RequestHandler` in `@sveltejs/kit` takes three optional generic parameters:

```ts
type RequestHandler<
    Params = Partial<Record<string, string>>,
    OutputBody = unknown,
    InputBody = unknown
> = (event: RequestEvent<Params>) => Response | Promise<Response>;
```

- **`Params`** — the path parameters inferred from the route pattern. For `src/routes/api/post/[id]/+server.ts`, this is `{ id: string }`.
- **`OutputBody`** — the shape of the data your handler returns via `json()`. Not structurally enforced by SvelteKit, but useful for self-documentation.
- **`InputBody`** — the shape of the request body you expect. Also self-documentation; runtime validation is your job.

You rarely spell all three out. The more common pattern in April 2026 is to let SvelteKit generate per-route types:

```ts
// src/routes/api/post/[id]/+server.ts
import type { RequestHandler } from './$types';
import { json, error } from '@sveltejs/kit';

export const GET: RequestHandler = async ({ params }) => {
    // params.id is known to be string here
    return json({ id: params.id });
};
```

The `./$types` module is generated by SvelteKit and knows about the `[id]` segment. It is the single best reason to keep your handlers in route folders and typed from the generated helper.

### 1.3 Typing the request body

For `POST` handlers you read the body from `request.json()`, which returns `Promise<unknown>`. You *must* validate before narrowing:

```ts
import * as v from 'valibot';
import { json, error } from '@sveltejs/kit';
import type { RequestHandler } from './$types';

const createPost = v.object({
    title: v.pipe(v.string(), v.minLength(1)),
    body: v.pipe(v.string(), v.minLength(1))
});

export const POST: RequestHandler = async ({ request }) => {
    const raw: unknown = await request.json().catch(() => null);
    const result = v.safeParse(createPost, raw);
    if (!result.success) error(400, result.issues[0]?.message ?? 'Invalid body');
    // result.output is now typed as { title: string; body: string }
    const { title, body } = result.output;
    return json({ id: crypto.randomUUID(), title, body }, { status: 201 });
};
```

`v.safeParse` returns `{ success: true, output }` or `{ success: false, issues }`. We read the first issue for a quick error message; a real API would return all of them.

### 1.4 Typing the response

For responses, the cleanest pattern is to define a response interface *in the same file*, use it inside the handler, and **re-export it**:

```ts
export interface PostCreatedResponse {
    readonly id: string;
    readonly title: string;
    readonly body: string;
}

export const POST: RequestHandler = async ({ request }) => {
    // ...
    const response: PostCreatedResponse = { id, title, body };
    return json(response, { status: 201 });
};
```

Now any consumer that imports `PostCreatedResponse` (in tests, in a shared types file, or in a fetch wrapper) has the exact shape the server is sending. This is the `+server.ts` version of "one source of truth" — it does not happen automatically the way remote functions do it, but the discipline is the same.

### 1.5 When types lie

Remember: **TypeScript cannot enforce a runtime contract with the outside world**. If you cast without validating — `const body = (await request.json()) as CreatePost;` — you have lied to the compiler about a value you do not control. Casts are debt, and in a handler that debt gets called in exactly as you would expect a stranger on the internet to call it. Always use a schema.

## 3. Interact — Make the compiler work for you

Write a handler with `export const POST = async (event) => ...`, no type annotation. Notice that `event.params.id` is `string | undefined`. Add `: RequestHandler` (imported from `./$types`) and suddenly it is just `string`, because the route pattern carries `[id]`. This is the single most useful reason to use the generated types.

<summary><strong>Q3.</strong> Does TypeScript protect you from a malicious request body?</summary>

No. Types are a compile-time contract. Runtime data can have any shape. A validator (Valibot) is the only protection.

So consumers can import it and type their `fetch` calls against the same shape the server returns. It is manual end-to-end typing — remote functions automate this.

## 6. Common mistakes

- **Importing `RequestHandler` from `@sveltejs/kit` in a route file.** Prefer `./$types` — it knows about the route's path segments.
- **Casting <code>request.json()</code> instead of validating.** A cast is debt; a schema is a fence.
- **Exporting the wrong thing.** `export const get` (lowercase) does nothing. It must be `export const GET`.
- **Returning a plain object from the handler.** You must return a `Response`. Use `json(value)` to wrap it.
