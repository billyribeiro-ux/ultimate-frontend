# Lesson 16.7 — Database in SvelteKit load functions

> **Atomic lesson format** — every lesson in this course follows the same four parts: **Concept**, **Style it**, **Interact**, **Mini-build**.

## 1. Concept — Connecting the database to your pages

### 1.1 What the problem is

You know how to query the database (Lessons 16.5–16.6). You know how SvelteKit load functions work (Module 9a). Now you need to connect them: when a user visits a page, the load function queries the database and the page component displays the results. The challenge is doing this with full type safety — the page component should know exactly what shape the data has, without any manual interface definitions that can drift.

### 1.2 The +page.server.ts load function pattern

SvelteKit's `+page.server.ts` files run exclusively on the server. They are the perfect place for database queries because:

1. They can import from `$lib/server/` (browser code cannot)
2. They run before the page renders (the component receives data as props)
3. Their return type flows through to the component automatically

```typescript
// src/routes/notes/+page.server.ts
import { db } from '$lib/server/db';
import { notes } from '$lib/server/db/schema';
import { eq, desc } from 'drizzle-orm';
import type { PageServerLoad } from './$types';

export const load: PageServerLoad = async () => {
    const allNotes = db.query.notes.findMany({
        orderBy: [desc(notes.createdAt)],
        with: { user: true }
    });

    return { notes: allNotes };
};
```

The `PageServerLoad` type is auto-generated by SvelteKit based on your route's parameters and parent layouts. It ensures your load function returns the correct shape.

### 1.3 Consuming load data in the component

On the component side, the data arrives through `$props()`:

```svelte
<script lang="ts">
    let { data } = $props();
    // data.notes is fully typed — each note has id, title, content, user, etc.
</script>

{#each data.notes as note (note.id)}
    <article>
        <h2>{note.title}</h2>
        <p>by {note.user.name}</p>
    </article>
{/each}
```

The type of `data` is inferred from what the load function returns. If you add a new field to the return object in `+page.server.ts`, it immediately appears in the component's `data` type. If you remove a field, TypeScript shows an error in the component. This is the typed end-to-end flow that Drizzle + SvelteKit provides.

### 1.4 Handling empty results and not-found states

Not every query returns data. A specific note might not exist:

```typescript
// src/routes/notes/[id]/+page.server.ts
import { error } from '@sveltejs/kit';
import type { PageServerLoad } from './$types';

export const load: PageServerLoad = async ({ params }) => {
    const noteId = Number(params.id);
    if (Number.isNaN(noteId)) {
        error(400, 'Invalid note ID');
    }

    const note = db.query.notes.findFirst({
        where: eq(notes.id, noteId),
        with: { user: true, notesTags: { with: { tag: true } } }
    });

    if (!note) {
        error(404, 'Note not found');
    }

    return { note };
};
```

Calling `error()` from `@sveltejs/kit` immediately stops execution and shows the nearest error page. The component code never runs when error() is called, so you can safely assume `data.note` exists after this pattern.

### 1.5 The $lib/server/ import boundary

The path `$lib/server/db` is enforced by SvelteKit's build system. If any file that runs in the browser (a `+page.svelte` file, a file imported by one, or a shared utility not under `$lib/server/`) tries to import from `$lib/server/`, the build fails with:

```
Cannot import $lib/server/db/index.ts into client-side code
```

This boundary is your safety net. Even if you accidentally write `import { db } from '$lib/server/db'` in a `.svelte` file, the error catches it instantly. Database connections, credentials, and queries physically cannot reach the browser.

### 1.6 Performance considerations

SvelteKit load functions run on every navigation to the page. For server-side rendered pages, they also run on the initial request. Some tips:

- **Query only what you need.** Use `columns` in `findMany` to select specific fields instead of loading entire rows with all columns.
- **Add limits.** Do not load 10,000 notes — add `.limit(50)` or implement pagination.
- **Use the relational API wisely.** Loading deeply nested relations (notes → tags → categories → departments) generates complex JOINs. Only load one or two levels deep.
- **Let SQLite do the work.** Filtering and sorting in SQL (`where`, `orderBy`) is faster than fetching all rows and filtering in JavaScript.

## 3. Interact — The typed data contract between server and client

The TypeScript concept is **inferred data contracts**. SvelteKit generates types that flow from your load function's return value through to the component's props. This forms a contract: the server promises a specific data shape, and the client depends on that shape.

```typescript
// Server side — the promise:
export const load: PageServerLoad = async () => {
    return {
        notes: db.query.notes.findMany({ with: { user: true } }),
        totalCount: db.select().from(notes).all().length
    };
};

// Client side — the dependency:
let { data } = $props();
// TypeScript knows: data.notes is the relational result type
// TypeScript knows: data.totalCount is number
```

If you change the server return (rename `notes` to `items`), TypeScript immediately shows errors in the component where `data.notes` is accessed. If you add a field, the component instantly has access. The contract is always in sync because it is derived from a single source — the load function's actual return statement.

This is fundamentally different from REST APIs where you write separate request/response types that can drift apart. With SvelteKit + Drizzle, the types flow from schema → query → load → component without a single manual type annotation on the data itself.

## 5. Check your understanding

<details>
<summary><strong>Q1.</strong> Why must database queries live in <code>+page.server.ts</code> rather than <code>+page.ts</code>?</summary>

`+page.ts` load functions run both on the server (during SSR) and in the browser (during client-side navigation). Database packages like `better-sqlite3` only work in Node.js — they would crash in the browser. `+page.server.ts` load functions run exclusively on the server, making it safe to import and use database code.
</details>

<details>
<summary><strong>Q2.</strong> How does the component know the type of <code>data.notes</code> without you writing an interface?</summary>

SvelteKit generates a `./$types` module for each route based on the load function's return type. When you type `export const load: PageServerLoad`, SvelteKit uses TypeScript's type inference to determine what the function returns and makes that type available as the `data` prop type in the corresponding `+page.svelte` component. The chain is: load return → generated type → component prop.
</details>

<details>
<summary><strong>Q3.</strong> What does calling <code>error(404, 'Note not found')</code> do in a load function?</summary>

It immediately throws a special SvelteKit error that halts load function execution and triggers the nearest `+error.svelte` page with a 404 status code. The component's code never runs — the user sees the error page instead. This is the correct pattern for handling "resource does not exist" cases.
</details>

<details>
<summary><strong>Q4.</strong> If your load function returns 10,000 notes, what is the performance problem and how do you fix it?</summary>

All 10,000 notes are serialized into the HTML response (for SSR) or the JSON payload (for client navigation). This creates a massive page that is slow to transfer, slow to parse, and slow to render. Fix it by adding `.limit(50)` to the query and implementing pagination — either page-based (offset/limit) or cursor-based (WHERE id > lastSeenId LIMIT 50).
</details>

<details>
<summary><strong>Q5.</strong> A student writes <code>import { db } from '$lib/server/db'</code> inside their +page.svelte file. What happens?</summary>

SvelteKit's build system detects the illegal import and throws a compile-time error: "Cannot import $lib/server/db/index.ts into client-side code." The dev server shows this error in the browser overlay and the build fails. The student must move the database query into `+page.server.ts` and access the results through the `data` prop.
</details>

## 6. Common mistakes

- **Returning non-serializable data from load.** Load function return values are serialized (converted to JSON for client-side navigation). Dates become strings, functions disappear, and class instances lose their methods. Always return plain objects and arrays. Drizzle results are already plain objects, so this is rarely an issue unless you attach custom methods.
- **Doing heavy computation in every load call.** Load functions run on every page visit. If you compute expensive aggregations (counting all notes, calculating statistics), consider caching the result or computing it less frequently. For this course's scale it does not matter, but in production it will.
- **Forgetting to handle the empty state.** If `db.query.notes.findMany()` returns an empty array, your component should show a helpful message ("No notes yet — create one!") rather than rendering an empty list with no explanation.
- **Querying inside the component with an `$effect`.** Some students try to fetch data in an `$effect` inside the .svelte file. This skips SSR (the page renders empty, then fills in on the client), requires a separate API endpoint, and loses type safety. Always use the load function pattern for initial page data.
