# Lesson 23.2 — Token pipeline

> **Atomic lesson format** — every lesson in this course follows the same four parts: **Concept**, **Style it**, **Interact**, **Mini-build**.

## 1. Concept — From design decisions to running CSS

### 1.1 The problem: designers and developers speak different languages

A designer opens Figma and sets a button color to "Brand Blue" with specific lightness, saturation, and hue values. They define spacing as multiples of 4px. They choose a type scale based on a 1.25 ratio. These decisions exist in Figma as layer properties, styles, and variables.

A developer opens VS Code and writes CSS. They need these same values as `var(--color-brand)`, `var(--space-md)`, and `var(--text-lg)`. If the developer manually transcribes values from Figma into CSS, two problems arise: (1) transcription errors — a designer changes the brand blue in Figma, but the CSS is not updated; (2) inconsistency — different developers transcribe slightly different values.

A **token pipeline** automates this translation. Tokens are defined once (in JSON, in Figma, in a design tool) and automatically transformed into the format each platform needs — CSS custom properties for the web, Swift constants for iOS, Kotlin constants for Android.

### 1.2 The Design Token Community Group (DTCG) format

The W3C Design Token Community Group has standardized a JSON format for design tokens:

```json
{
    "color": {
        "brand": {
            "$value": "oklch(65% 0.22 270)",
            "$type": "color",
            "$description": "Primary brand color — blue-violet"
        },
        "surface": {
            "$value": "oklch(98% 0.01 270)",
            "$type": "color",
            "$description": "Default page background"
        }
    },
    "space": {
        "md": {
            "$value": "clamp(1rem, 3vw, 1.5rem)",
            "$type": "dimension",
            "$description": "Medium spacing — general-purpose gap"
        }
    }
}
```

The `$value` is the token's value. The `$type` declares what kind of token it is (color, dimension, duration, etc.). The `$description` documents the token's purpose. This format is tool-agnostic: any pipeline tool can read it.

### 1.3 Style Dictionary: the industry-standard pipeline

Style Dictionary (by Amazon) is the most widely used token pipeline tool. It reads token JSON files, applies transforms (converting color formats, adding platform-specific prefixes), and outputs platform-specific files:

```
tokens.json → Style Dictionary → css/variables.css
                                → ios/Colors.swift
                                → android/colors.xml
```

For a Svelte project, you only need the CSS output. Style Dictionary can generate an `@layer tokens` block with all your CSS custom properties, organized by category.

### 1.4 The Cobalt pattern: tokens in TypeScript

An alternative approach, gaining popularity in 2025-2026, uses TypeScript files as the token source. The Cobalt pipeline (`@cobalt-ui/core`) reads a `tokens.json` conforming to the DTCG format and generates typed CSS and TypeScript output. The advantage is type safety: if a token references another token that does not exist, the pipeline fails at build time.

```typescript
// Generated by Cobalt — do not edit manually
export const colorBrand = 'var(--color-brand)' as const;
export const spaceMd = 'var(--space-md)' as const;
```

These constants can be imported in Svelte components for type-safe token references, though in practice, using CSS custom properties directly in `<style>` blocks is simpler and more idiomatic.

### 1.5 Multi-theme token sets

A design system with multiple themes (light, dark, brand-A, brand-B) defines a **base token set** and **theme overrides**:

```json
{
    "color": {
        "surface": {
            "$value": "oklch(98% 0.01 270)",
            "$type": "color",
            "$extensions": {
                "themes": {
                    "dark": "oklch(18% 0.02 270)",
                    "brand-warm": "oklch(97% 0.02 50)"
                }
            }
        }
    }
}
```

The pipeline generates a CSS file with the base tokens on `:root` and theme-specific overrides on `[data-theme="dark"]`, `[data-theme="brand-warm"]`, etc. Applying a theme is as simple as setting `data-theme` on the `<html>` element — all components pick up the new values automatically because they reference `var(--color-surface)`, not the raw OKLCH value.

### 1.6 Connecting Figma to the pipeline

Figma Variables (introduced in 2023, mature by 2026) map directly to design tokens. A Figma plugin like "Tokens Studio" or Figma's built-in REST API exports variables as JSON in the DTCG format. A CI job runs the pipeline tool on every push to the design tokens repository, generating CSS and publishing it as an npm package.

The flow: Figma Variables → JSON export → Style Dictionary/Cobalt → CSS custom properties → npm package → consumed by SvelteKit app.

This automation ensures that when a designer changes a color in Figma, the CSS updates automatically (after a PR review and merge), and no developer manually transcribes values.

### 1.7 "In Production" — token pipeline at a multi-brand company

A company operating three brands (each with different colors, typography, and personality) used a single SvelteKit codebase for all three products. They defined three token sets: `brand-alpha`, `brand-beta`, `brand-gamma`. The token pipeline generated three CSS files. At build time, an environment variable selected the active brand. Each brand's token CSS was imported into `app.css`, overriding the base tokens. One codebase, three visual identities, zero component code duplication. When the brand-alpha designer changed the primary color in Figma, the pipeline generated new CSS, a PR was auto-created, and the change was live in production within an hour.

### 1.8 The TypeScript angle

The token pipeline itself can be typed:

```typescript
interface DesignToken {
    $value: string;
    $type: 'color' | 'dimension' | 'duration' | 'fontFamily' | 'fontWeight' | 'number';
    $description?: string;
    $extensions?: {
        themes?: Record<string, string>;
    };
}

interface TokenGroup {
    [key: string]: DesignToken | TokenGroup;
}

function isToken(node: DesignToken | TokenGroup): node is DesignToken {
    return '$value' in node;
}
```

The `isToken` type guard distinguishes between a token (has `$value`) and a group (contains nested tokens or groups). This enables recursive traversal of the token tree with full type safety.

### 1.9 Common interview question

**Q: "Why would you use a token pipeline instead of just writing CSS custom properties by hand?"**

**Model answer:** A token pipeline automates the translation from design decisions (in Figma, JSON, or TypeScript) to platform-specific output (CSS, Swift, Kotlin). This eliminates manual transcription errors, ensures the source of truth is the design tool (not scattered CSS files), enables multi-platform support from a single source, supports multi-theme generation automatically, and provides a reviewable change trail. For a single-developer project, manual CSS is fine. For a team with designers and multiple platforms, the pipeline pays for itself quickly.

## Deep Dive

**Token resolution and aliasing.** Tokens can reference other tokens via aliases: `"color-brand-dim": { "$value": "{color.brand}", "$type": "color" }`. The pipeline resolves these references during build, replacing aliases with concrete values. This enables semantic layering: `color-brand-dim` is "a darker version of the brand color," and if the brand color changes, `color-brand-dim` updates automatically.

**Token versioning.** When tokens change, the CSS output changes, which can break visual tests. Versioning tokens (using semver alongside the component library) lets teams pin to a specific token version and upgrade intentionally. A new shade of brand-blue is a minor version bump. Removing a token is a major version bump.

**Custom transforms.** Style Dictionary allows custom transforms — functions that modify token values during pipeline execution. A common transform converts hex colors to OKLCH, adds `clamp()` wrappers to spacing values, or generates CSS `@media` blocks for responsive tokens. Custom transforms are how you enforce PE7 conventions: "all colors must be OKLCH" is a transform that converts any non-OKLCH color to OKLCH format.

**Token documentation generation.** The pipeline can generate documentation alongside CSS — an HTML page showing every token with its name, value, visual preview, and description. This becomes the "tokens" section of the design system documentation (Lesson 23.4). Automated documentation ensures that docs are always in sync with the actual token values.

<details>
<summary><strong>Q1.</strong> What is the DTCG format for design tokens?</summary>

The Design Token Community Group format is a standardized JSON structure for defining design tokens. Each token has a `$value` (the actual value), `$type` (color, dimension, duration, etc.), and optional `$description`. The format is tool-agnostic, meaning any pipeline tool can read and process it.
</details>

<details>
<summary><strong>Q2.</strong> How does a token pipeline eliminate transcription errors between design and development?</summary>

The pipeline reads tokens directly from the design tool's output (Figma export, JSON file) and generates CSS automatically. Developers never manually type color values or spacing numbers. When a designer changes a value, the pipeline regenerates the CSS, ensuring the design and code are always in sync.
</details>

<details>
<summary><strong>Q3.</strong> How do multi-theme token sets work with CSS custom properties?</summary>

Base tokens are defined on `:root`. Theme overrides are defined on attribute selectors like `[data-theme="dark"]`. When the `data-theme` attribute changes on the `<html>` element, the overridden custom properties take precedence via CSS specificity. Components reference `var(--color-surface)` regardless of theme — the value changes automatically based on the active theme.
</details>

<details>
<summary><strong>Q4.</strong> What is the advantage of token aliasing?</summary>

Aliases let tokens reference other tokens: `color-brand-dim` references `color-brand`. If the brand color changes, all aliases update automatically during pipeline execution. This enables semantic layering — tokens describe relationships (brand-dim is "a darker version of brand") rather than independent values.
</details>

<details>
<summary><strong>Q5.</strong> When is a token pipeline not necessary?</summary>

For a single-developer project with one theme and one platform (web only), writing CSS custom properties by hand is simpler and has no build tooling overhead. The pipeline becomes valuable when multiple people maintain tokens, when tokens must generate output for multiple platforms, or when multi-theme support is required.
</details>

## 6. Common mistakes

- **Hardcoding theme values in component styles instead of using tokens.** If a component uses `oklch(18% 0.02 270)` instead of `var(--color-surface)`, it will not respond to theme changes. Every visual value must go through the token system.
- **Naming tokens by visual value instead of purpose.** `--color-blue-500` is a visual name that becomes misleading if the brand color changes to green. `--color-brand` describes purpose and remains accurate regardless of the value.
- **Not validating token JSON before pipeline execution.** Invalid JSON or missing required fields (`$value`, `$type`) should fail the pipeline with a clear error. Without validation, the pipeline might silently skip invalid tokens and produce incomplete CSS.
- **Manually editing generated CSS.** If the pipeline generates `tokens.css`, editing it manually means your changes will be overwritten on the next pipeline run. Always edit the source JSON and regenerate.
