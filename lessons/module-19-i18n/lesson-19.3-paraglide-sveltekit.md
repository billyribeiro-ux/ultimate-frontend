# Lesson 19.3 — Paraglide.js with SvelteKit

> **Atomic lesson format** — every lesson in this course follows the same four parts: **Concept**, **Style it**, **Interact**, **Mini-build**.

## 1. Concept — Compile-time i18n that thinks like Svelte

### 1.1 The problem: runtime translation lookup is wasteful

Most i18n libraries work like this: they load a JSON dictionary into memory, and every time your template renders a translated string, the library searches the dictionary by key. This means every page ships the entire dictionary for every locale — even messages used only on the settings page arrive in the bundle for the home page. In a large application with 500 messages across three locales, you ship 1,500 strings when only 50 are needed on any given page.

The lookup itself is fast (it is just a property access on an object), but the bundle cost is not. On a mobile connection, an extra 40 KB of unused translations adds noticeable time to interactive. And because the lookup is a runtime operation, TypeScript cannot verify at build time that every key exists or that you passed the correct variables.

### 1.2 How Paraglide solves it

Paraglide.js, created by the Inlang project, takes a fundamentally different approach. It is a **compiler** for translations. During your build, Paraglide reads your message files, and for each message key, generates a JavaScript function:

```typescript
// Generated by Paraglide at build time
export function pricing_title(): string {
  return "Choose your plan";
}

export function pricing_features(params: { count: number }): string {
  // ICU plural logic compiled to a switch statement
  const { count } = params;
  if (count === 0) return "No features included";
  if (count === 1) return "1 feature included";
  return `${count} features included`;
}
```

Each locale gets its own set of functions. At runtime, Paraglide loads only the current locale's module. Vite's tree-shaking removes any message functions that are not imported by the current page. The result: your bundle contains exactly the strings the page needs, in exactly one locale, with zero lookup overhead.

### 1.3 Setting up Paraglide in SvelteKit

Installation with pnpm:

```bash
pnpm add @inlang/paraglide-sveltekit
pnpm exec paraglide-js init --project ./project.inlang
```

This creates a `project.inlang/` directory with a `settings.json` that declares your source locale and target locales:

```json
{
  "sourceLanguageTag": "en",
  "languageTags": ["en", "pt-BR", "ar"],
  "modules": [
    "https://cdn.jsdelivr.net/npm/@inlang/message-lint-rule-empty-pattern/dist/index.js",
    "https://cdn.jsdelivr.net/npm/@inlang/message-lint-rule-missing-translation/dist/index.js"
  ]
}
```

Message files live in `messages/` as JSON. The English source file:

```json
// messages/en.json
{
  "greeting": "Welcome, {name}!",
  "item_count": "{count, plural, =0 {No items} one {# item} other {# items}}"
}
```

The SvelteKit adapter is configured in `vite.config.ts`:

```typescript
import { paraglide } from '@inlang/paraglide-sveltekit/vite';

export default defineConfig({
  plugins: [
    paraglide({
      project: './project.inlang',
      outdir: './src/lib/paraglide'
    }),
    sveltekit()
  ]
});
```

After running the build or dev server, Paraglide generates typed functions in `src/lib/paraglide/messages.ts`:

```typescript
import * as m from '$lib/paraglide/messages';

// In a Svelte component:
m.greeting({ name: 'Ada' }); // "Welcome, Ada!"
m.item_count({ count: 3 });  // "3 items"
```

### 1.4 How it connects to what you already know

Paraglide's compilation step is philosophically identical to what the Svelte compiler does with your `.svelte` files. Svelte compiles declarative templates into imperative DOM operations. Paraglide compiles declarative message patterns into imperative string-building functions. Both eliminate runtime overhead by moving work to build time. Both generate TypeScript types automatically. Both produce output that Vite can tree-shake.

The `$lib/paraglide/messages` import path follows the same `$lib` convention you have used since Module 1. The generated code is plain TypeScript — no magic, no hidden state, no framework coupling.

### 1.5 Server-side rendering and locale detection

In SvelteKit, pages render first on the server. Paraglide's SvelteKit adapter handles this by reading the locale from the URL (which you will configure in Lesson 19.4) and setting the active language before each server render. On the client, it reads the locale from the URL and activates the matching message module.

The adapter adds a `handle` hook to your `hooks.server.ts` that sets the locale for each request:

```typescript
import { i18n } from '$lib/paraglide/i18n';
import type { Handle } from '@sveltejs/kit';

export const handle: Handle = i18n.handle();
```

This ensures that SSR produces the correct locale's HTML, which is critical for SEO (search engines crawl the server-rendered HTML) and for avoiding a flash of untranslated content on first load.

### Deep Dive — How Paraglide tree-shakes per page and per locale

Paraglide's tree-shaking operates at two levels, and understanding both is essential for keeping your i18n bundle minimal.

**Per-locale tree-shaking:** Each locale's messages are generated as a separate ES module. When the active locale is `en`, only `messages/en.js` is imported. The `pt-BR` and `ar` modules are not part of the bundle. When the user switches locale, a dynamic `import()` loads the new locale's module. This means your initial page load only pays for one locale's strings.

**Per-page tree-shaking:** Because each message is a named export, Vite's tree-shaker can determine which messages are actually imported on each page. If the pricing page imports `m.pricing_title` and `m.pricing_features` but not `m.greeting`, the greeting function is excluded from the pricing page's chunk. This is why Paraglide generates individual functions rather than a single dictionary object — a dictionary cannot be partially tree-shaken.

The combination of both levels produces remarkably small bundles. A page that uses 10 messages in one locale might ship only 800 bytes of translated strings, compared to 40 KB for a runtime library that ships the full dictionary.

There is one caveat: if you import all messages with `import * as m from '$lib/paraglide/messages'` and then access them dynamically (`m[key]()`), Vite cannot tree-shake any messages because it cannot determine which keys are accessed at build time. Always use direct named imports when possible: `import { pricing_title, pricing_features } from '$lib/paraglide/messages'`. The `* as m` pattern is acceptable in small applications but degrades tree-shaking in large ones.

Paraglide also supports lazy loading of locale modules. When a user switches from English to Portuguese, the Portuguese module is loaded via `import()`. If you preload locale modules for common secondary locales (e.g., `<link rel="modulepreload" href="/pt-BR-messages.js">`), the switch feels instant.

## 3. Interact — simulating Paraglide message compilation

The problem: demonstrating compile-time message generation in a browser-based mini-build requires simulating the compilation step.

The solution builds a simplified compiler in TypeScript. The student enters a message pattern and the component generates the equivalent JavaScript function:

```typescript
function compileMessage(key: string, pattern: string): string {
  const params = extractParams(pattern);
  const paramList: string = params.length > 0
    ? `params: { ${params.map(p => `${p.name}: ${p.type}`).join('; ')} }`
    : '';
  const body: string = compilePattern(pattern);
  return `export function ${key}(${paramList}): string {\n  ${body}\n}`;
}
```

The compiled output updates reactively as the student modifies the input pattern.

## 5. Check your understanding — 5 questions

<details>
<summary><strong>Q1.</strong> How does Paraglide differ from runtime i18n libraries like svelte-i18n?</summary>

Paraglide compiles message patterns into JavaScript functions at build time. Each message becomes a named export that Vite can tree-shake. Runtime libraries load a dictionary object at runtime and look up keys dynamically, shipping all messages for a locale regardless of which page uses them. Paraglide eliminates lookup overhead and reduces bundle size by including only the messages each page imports.
</details>

<details>
<summary><strong>Q2.</strong> Why does Paraglide generate individual named exports instead of a single dictionary object?</summary>

Individual named exports enable Vite's tree-shaker to remove unused messages from the bundle. If only 10 of 500 messages are imported on a page, the other 490 are excluded. A dictionary object cannot be partially tree-shaken because Vite cannot prove which keys are accessed at runtime.
</details>

<details>
<summary><strong>Q3.</strong> What role does the handle hook in hooks.server.ts play in Paraglide's SvelteKit integration?</summary>

The handle hook reads the locale from the incoming request (typically from the URL path prefix) and sets the active language before the server renders any component. This ensures SSR produces HTML in the correct locale, which is critical for SEO crawlers and prevents a flash of untranslated content on initial page load.
</details>

<details>
<summary><strong>Q4.</strong> What happens if you use `import * as m` and then access messages dynamically with `m[key]()`?</summary>

Vite cannot determine which message keys are accessed at build time, so it includes all messages in the bundle, defeating tree-shaking. For large applications, this can add tens of kilobytes of unused strings. Use direct named imports (`import { greeting } from '$lib/paraglide/messages'`) when possible.
</details>

<details>
<summary><strong>Q5.</strong> Where should message JSON files live in a Paraglide SvelteKit project, and how are generated files consumed?</summary>

Message JSON files live in the `messages/` directory (one file per locale, e.g., `messages/en.json`, `messages/pt-BR.json`). Paraglide's Vite plugin reads these files and generates typed TypeScript functions in the `outdir` specified in the Vite config (typically `src/lib/paraglide/messages.ts`). Components import from `$lib/paraglide/messages` like any other module.
</details>

## 6. Common mistakes — 4 pitfalls

1. **Forgetting to add the Paraglide Vite plugin before sveltekit().** The `paraglide()` plugin must be listed before `sveltekit()` in the `plugins` array so that message files are generated before SvelteKit processes imports to `$lib/paraglide/`.

2. **Editing generated files in src/lib/paraglide/.** The files in the `outdir` are regenerated on every build. Edits are overwritten. Always edit the source message JSON files in `messages/` and let Paraglide regenerate.

3. **Missing the handle hook.** Without the `i18n.handle()` in `hooks.server.ts`, server-rendered pages always use the default locale regardless of the URL. The client-side locale switch works but the initial HTML is wrong, causing content flashes.

4. **Not running `paraglide-js init`.** The Inlang project configuration (`project.inlang/settings.json`) must exist before Paraglide can generate anything. The `init` command creates it with sensible defaults.
